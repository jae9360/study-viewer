1과목. 요구사항 확인

소프트웨어 생명주기(SDLC) 모델

소프트웨어의 개발~폐기 전 과정을 단계화한 프로세스 모델. 타당성 검토→계획→요구분석→설계→구현→테스트→운영/유지보수.

[폭프나반] 생명주기 모델 종류 — 폭포수 / 프로토타이핑 / 나선형 / 반복적
모델핵심
폭포수(Waterfall)순차 진행·후퇴 불가, 요구사항 명확할 때, 문서(산출물) 중심
프로토타이핑시제품으로 고객 피드백 → 요구사항 구체화, 요구 불명확 시 유리
나선형(Spiral)폭포수+프로토타입+위험 분석, 반복적 점진 개발, 대규모·고위험에 적합
반복적(Iteration)증분/진화 방식으로 나누어 릴리즈, 병렬 개발 가능
[계위개고] 나선형 모델 절차 — 계획 및 정의 → 위험 분석 → 개발 → 고객 평가 (반복)
나선형 모델은 "위험 분석"이 답의 핵심 키워드. "위험을 관리·최소화하며 점진적으로 개발하는 모델은?" 유형과 절차 4단계를 순서대로 쓰는 문제가 반복 출제.

애자일(Agile) 방법론

유연하고 빠르게 개발하는 방법론 절차·문서보다 소통과 동작하는 SW, 계약·계획보다 고객 협업·변화 대응을 중시(애자일 선언 4가치). 짧은 반복 주기로 개발.

애자일 유형 (XP · 스크럼 · 린 · 칸반)
  • XP: 1~3주 릴리즈. 5가치=용기·단순성·의사소통·피드백·존중. 실천: 짝 프로그래밍, 공동 코드 소유, 지속적 통합(CI), TDD, 리팩토링
  • 스크럼: 백로그·스프린트(2~4주)·일일 스크럼·PO/스크럼 마스터·번다운 차트
  • 린(Lean): 도요타 생산방식, 낭비 제거 7원칙
  • 칸반: 작업 흐름 시각화(보드), WIP 제한

객체 지향

[클객 메 메인속] 구성요소 (메메한 인간의 속내) — 클래스 / 객체 / 메서드 / 메시지 / 인스턴스 / 속성
요소정의
클래스유사 객체의 공통 특성(속성+메서드)을 묶은 틀·설계도
객체속성(데이터)과 메서드(함수)를 묶은 SW 모듈
메서드객체가 수행하는 연산·동작
메시지객체 간 상호작용 수단(메서드 호출 요청)
인스턴스클래스로부터 실제 생성된 객체
속성객체가 가진 데이터·상태
[캡상다추정관] 객체 지향 기법 — 캡슐화 / 상속성 / 다형성 / 추상화 / 정보은닉 / 관계성
  • 캡슐화: 속성+메서드를 하나로 묶고 외부 접근 제한 (결합도↓ 응집도↑)
  • 상속성: 상위 클래스 속성·메서드를 하위가 물려받아 재사용
  • 다형성: 하나의 메시지에 객체마다 다른 동작 — 오버로딩(같은 이름·다른 매개변수) / 오버라이딩(상위 메서드 재정의)
  • 추상화: 공통 핵심 성질만 추출해 모델링
  • 정보은닉: 내부 구현 은폐, 인터페이스만 공개 (사이드 이펙트 최소화)
  • 관계성: 객체 간 연관·집합·일반화 관계 표현
오버로딩 vs 오버라이딩 구분 서술이 단골. 오버로딩 = 같은 이름, 다른 매개변수(개수·타입) / 오버라이딩 = 상속받은 메서드를 재정의. "다형성"이라는 용어 자체를 쓰게 하는 단답형도 출제.
[SOLID] 객체지향 설계 원칙 — SRP 단일 책임 / OCP 개방 폐쇄 / LSP 리스코프 치환 / ISP 인터페이스 분리 / DIP 의존성 역전
원칙내용
SRP클래스는 하나의 책임만 가진다
OCP확장에 열려 있고 변경에 닫혀 있어야 한다
LSP하위 클래스는 상위 클래스를 대체 가능해야 한다
ISP사용하지 않는 인터페이스에 의존하지 않는다
DIP구체가 아닌 추상(인터페이스)에 의존한다
설명을 주고 영문 약어(SRP·OCP·LSP·ISP·DIP) 또는 한글 명칭을 쓰게 하는 단답형 최빈출. 특히 OCP(확장 개방·변경 폐쇄)와 ISP가 자주 출제.
객체 지향 분석 방법론 종류
  • 럼바우(OMT): 그래픽 표기 모델링 (아래 [객동기])
  • 부치(Booch): 미시적+거시적 프로세스, 클래스·객체 식별
  • 야콥슨(OOSE): 유스케이스 중심
  • 코드-요돈: E-R 다이어그램 활용
  • 워프스-브록: 분석·설계 구분 없이 연속 수행
[객동기] 럼바우 분석 절차 (격동기) — 객체 모델링(객체 다이어그램) → 동적 모델링(상태 다이어그램) → 기능 모델링(자료 흐름도 DFD)
럼바우는 순서(객→동→기)와 각 단계의 산출 다이어그램 매칭(객체=객체 다이어그램, 동적=상태 다이어그램, 기능=DFD)이 그대로 답으로 출제됨.

프로젝트 관리

비용산정 모형 종류
  • 하향식: 전문가 판단, 델파이 기법
  • LOC: 예측치 = (낙관치+4×기대치+비관치)÷6
  • Man Month: 1인 1개월 작업량 기준, 기간=MM÷인원
  • COCOMO(보헴): 조직형(5만 라인↓)/반분리형(30만↓)/임베디드형(30만↑)
  • Putnam: Rayleigh-Norden 곡선 기반, 도구=SLIM
  • FP(기능 점수): 기능별 가중치 산정, 도구=ESTIMACS
일정관리 모델 종류
  • CPM: 임계 경로(최장 경로)로 최소 소요 기간 산정
  • PERT: 낙관·기대·비관 3점 추정(신규 프로젝트)
  • CCPM: 자원 제약 고려 임계 연쇄
  • 간트 차트: 작업 일정 막대 시각화
[회전완수] 위험 대응 전략 — 회피(계획 변경) / 전가(제3자 이전·보험) / 완화(확률·영향 감소) / 수용(수용 후 대비책)
COCOMO는 조직형(5만 라인 이하) / 반분리형(30만 이하) / 임베디드형(30만 초과) 3유형과 경계값이 단답으로 출제. 위험 대응 4전략은 사례를 주고 전가(보험·외주)인지 완화인지 고르게 하는 유형.

뷰관점내용

프로세스개발자동시성·성능·스레드
구현(개발)개발자모듈·컴포넌트 구성
배포(물리)시스템 엔지니어HW에 SW 배치
아키텍처 패턴 (계층화·C/S·파이프필터·브로커·MVC)
  • 계층화: 계층 분리(OSI 모델)
  • 클라이언트-서버: 서비스 제공/요청 분리
  • 파이프-필터: 필터 처리→파이프 전달(UNIX 쉘)
  • 브로커: 분산 컴포넌트 통신 중재
  • MVC: Model(데이터)-View(화면)-Controller(제어)
[SACAA] 아키텍처 비용 평가 모델 (사카린) — SAAM(변경용이성·최초) / ATAM(품질속성 트레이드오프) / CBAM(경제성) / ADR(응집도) / ARID(ATAM+ADR, 부분 설계)

요구공학 4단계(도출→분석→명세→확인) 순서 쓰기와 워크스루/인스펙션 구분(비공식 검토 vs 공식 검토)이 출제 포인트.
현행 시스템 분석

현행 플랫폼 기능·구조, OS, 네트워크, DBMS, 비즈니스 융합을 파악해 개발 범위·방향 결정. DBMS 분석 고려사항: 가용성·성능·상호 호환성·기술 지원·구축 비용.

함수 종속 (완전·부분·이행)

X→Y: X(결정자)가 Y(종속자)를 결정. 완전=기본키 전체에 종속 / 부분=복합키 일부에만 종속 / 이행=X→Y, Y→Z ⇒ X→Z

정규화: 이상 현상 제거를 위해 릴레이션을 무손실 분해(중복 최소화·종속성 제거). 반정규화: 성능·운영 단순화를 위한 의도적 중복·통합(일관성 저하 위험).

[원부이 결다조] 정규화 단계 — 1NF 원자화 → 2NF 부분함수종속 제거 → 3NF 이행함수종속 제거 → BCNF 결정자가 후보키 → 4NF 다치종속 제거 → 5NF 조인종속 제거
3과목 최중요. "○○ 종속을 제거하면 몇 정규형?" 유형이 매회 출제 수준 — 특히 2NF(부분)·3NF(이행)·BCNF(결정자가 후보키). 이상 현상 3종(삽삭갱)의 명칭·정의 서술, 카디널리티=튜플 수 / 차수(Degree)=속성 수도 단답 단골.

4과목. 통합 구현

연계 메커니즘

송신 시스템 → 중계 시스템 → 수신 시스템. 직접 연계(DB Link·API·JDBC — 빠르지만 결합도↑) vs 간접 연계(EAI·ESB·웹서비스·소켓 — 유연하지만 성능↓).

주요 용어
  • 커넥션 풀: DB 연결을 미리 만들어 재사용 → 연결 생성 비용 절감
  • JDBC: Java의 DB 접속 표준 API
  • 소켓: 네트워크 통신 끝점, 포트 할당으로 클라이언트 연결 처리
  • API: 기능·데이터 접근 인터페이스

EAI / ESB

EAI: 기업 내 이기종 애플리케이션 간 정보 전달·연계·통합 솔루션.

[포허 메하] EAI 구축 유형 — 포인트 투 포인트 / 허브 앤 스포크 / 메시지 버스 / 하이브리드
유형특징
Point-to-Point1:1 직접 연결, 단순·저렴, 시스템 증가 시 연결 폭증
Hub & Spoke중앙 허브 경유, 확장 용이, 허브 장애 시 전체 영향
Message Bus(ESB)미들웨어(버스) 연계, 뛰어난 확장성·대용량
Hybrid그룹 내 허브앤스포크 + 그룹 간 메시지 버스
EAI는 유형 설명 → 이름 쓰기가 기출 패턴. "중앙 허브 장애 시 전체 영향" = 허브 앤 스포크, "미들웨어(버스) 활용·확장성 우수" = 메시지 버스로 키워드 매칭. ESB의 느슨한 결합(Loosely Coupled) 용어도 출제.

ESB: 서비스 중심·버스 기반으로 느슨한 결합(Loosely Coupled) 통합 미들웨어. 표준(웹서비스) 지향.

5과목. 인터페이스 구현

기술정의
JSON속성-값 쌍의 경량 데이터 교환 포맷. 언어 독립. {"name":"홍길동"}. 배열 []=선형 구조, 객체 {}=키-값
XMLHTML 한계·SGML 복잡성을 해결한 확장 가능 마크업 언어(사용자 정의 태그)
AJAX자바스크립트 비동기 통신 — 페이지 전체 새로고침 없이 일부 갱신(XMLHttpRequest)
REST자원(URI)+행위(HTTP Method: GET/POST/PUT/DELETE)+표현(JSON·XML). 준수 시 RESTful
DOMXML/HTML을 트리 객체로 접근·조작(전체 메모리 로딩) ↔ SAX(이벤트 기반 순차 파싱)
XSLTXML을 다른 XML/HTML로 변환하는 언어
HTML/CSS웹 문서 구조 마크업 / 표현(스타일) 정의
5과목은 용어 정의 → 영문 명칭 쓰기가 핵심. JSON(속성-값 쌍 경량 포맷), AJAX(비동기·일부만 갱신), REST(자원 URI + HTTP 메서드) 정의는 통째로 암기. REST 구성 3요소 자원/행위/표현도 출제됨.

  • 검증 도구: xUnit, STAF, FitNesse, NTAF, Selenium, watir
  • 전송 암호화: IPSec(IP 계층) / SSL·TLS(전송 계층) / S-HTTP(응용 계층)
